# Global cleanup after PWA migration

source : <https://trello.com/c/mfqJTEr0/37-pass-to-pwa>

## Purpose

We've made a lot of changes to migrate the app to PWA. The goal here is to take a critical look at our codebase and assess our choices. Is there unnecessary complexity, can we reduce duplication, is it easy to comprehend and what can we do better?

---

## Implementation Plan

### 1 — Rename `ConnectionIndicator` → `OfflineStatus`

The component no longer indicates server connection health — it shows offline state, pending mutation count, and sync progress. Rename component, file, and CSS module for clarity. Update `MainLayout` import accordingly.

---

### 2 — Factorize the three waypoint mutations in `useTracks.ts`

`useAddWaypoint`, `useEditWaypoint`, `useDeleteWaypoint` all follow the same skeleton:

1. Read `gpxContent` from the `['tracks', id]` cache
2. Apply a GPX transform function
3. Enqueue `PUT_TRACK_GPX`
4. `onSuccess`: update the cache with the new GPX, invalidate `['tracks', id, 'parsed']` and `['mutation-queue']`

Extract a `useGpxMutation(trackId, transform)` helper that encapsulates this pattern, reducing ~100 lines of duplication to three one-liners.

---

### 3 — Fix the reconnection flush trigger in `useNetworkSync`

**Current problem**: when coming back online, both the `'online'` DOM event and the `isReady` state change trigger a flush — the `'online'` event is redundant and `navigator.onLine` alone is not sufficient to know the server can accept mutations.

**Fix**:

- Remove `window.addEventListener('online', flush)` — keep only the `useEffect` on `isReady`
- When `'online'` fires, invalidate `['/health']` to force an immediate health re-check instead of waiting for the next poll interval
- This way the flush only happens once we have confirmation the server is ready, with minimal delay

---

### 4 — Disable offline-unsupported actions in the UI

The task file (PR6) called for gracefully disabling actions not yet supported offline. Not yet done:

- **"Upload new track" button** (`NewTrackModal`, `Tracks.tsx`) — disable and show tooltip when `!isReady`
- **`GpxUploader` on `Home.tsx`** — show an offline message instead of the uploader when `!isReady`

Destructive actions (delete track, delete trip, create trip) are handled by a separate ticket — they will fail with an error notification, which is acceptable for now.
